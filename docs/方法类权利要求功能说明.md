# 方法类权利要求拆解构图系统 — 功能说明与开发规范

> 版本：V1.0 | 日期：2026-06-18

---

## 一、功能概述

### 1.1 背景

当前 Patent2Pic 仅支持**结构类权利要求**（装置/设备/系统）的拆解构图，将部件节点和位置/包含/动作关系可视化为树状/网状拓扑图。

方法类权利要求（方法/流程/过程）具有完全不同的语义特征——以步骤为节点、以时序/分支/触发为关系，呈现为**控制流图/流程图**。本系统在现有架构上扩展，支持方法类权利要求的自动识别、拆解和构图。

### 1.2 核心目标

1. **自动识别**权利要求类型（结构类 vs 方法类）
2. **方法类专用提示词**引导 AI 提取步骤/判断/条件节点和顺序/分支/触发关系
3. **流程图可视化**：菱形判断、六边形条件、分支路径、反馈回路
4. **与结构类共用**底层引擎（X6、ELK、导出、翻译对齐），通过 `claimType` 路由切换

### 1.3 适用范围

- 纯方法类权利要求（如"一种数据处理方法"）
- 纯结构类权利要求（如"一种传动装置"）——已有功能
- 混合型权利要求（如"一种控制系统，其特征在于包括以下步骤"）——按主体判断类型

## 二、系统架构

### 2.1 数据流

```
用户输入权利要求文本
    ↓
predictClaimType(text) ──→ 预判 'method' | 'structure'
    ↓
buildMessages(claimText, providerType, claimType) ──→ 选择对应提示词
    ↓
AI 返回 JSON（含 claimType 字段确认）
    ↓
parseExtractResult() → validateExtractResult()
    ├─ 结构类：component/subsystem/feature + position/action/containment/logical/attribute
    └─ 方法类：step/decision/condition + sequence/branch_true/branch_false/trigger/feedback/parallel/attribute
    ↓
batchBuild(result, claimType)
    ├─ 结构类路径：现有逻辑
    └─ 方法类路径：自定义 shape + 固定 LR 布局 + feedback 边特殊处理
    ↓
渲染完成 → 交互/导出
```

## 三、类型系统设计

### 3.1 ClaimType（新增）

```typescript
export type ClaimType = 'structure' | 'method'
```

### 3.2 NodeType（扩展）

```typescript
export type NodeType =
  // 结构类
  | 'component'    // 部件
  | 'subsystem'    // 子系统
  | 'feature'      // 特征
  // 方法类
  | 'step'         // 执行步骤
  | 'decision'     // 逻辑判断（菱形，双出口）
  | 'condition'    // 触发条件（六边形，单出口）
```

### 3.3 RelationType（扩展）

```typescript
export type RelationType =
  // 结构类专属
  | 'position'       // 位置关系（"设置于"、"位于"）
  | 'action'         // 动作关系（"驱动"、"控制"）
  | 'containment'    // 包含关系（"包括"、"具有"）
  // 方法类专属
  | 'sequence'       // 先后顺序（"然后"、"接着"）
  | 'branch_true'    // 条件为真分支（decision → 是）
  | 'branch_false'   // 条件为假分支（decision → 否）
  | 'trigger'        // 触发关系（"当...时"）
  | 'feedback'       // 反馈回路（"返回"、"重复"）
  | 'parallel'       // 并行执行（"同时"、"并行"）
  // 共用
  | 'logical'        // 逻辑关系
  | 'attribute'      // 属性/参数（结构类：特性描述；方法类：参数值/阈值）
```

### 3.4 ExtractResult 扩展

```typescript
export interface ExtractResult {
  claimType: ClaimType        // 新增：权利要求类型
  claimId: string
  nodes: ExtractNode[]
  edges: ExtractEdge[]
  groups: ExtractGroup[]
  translatedClaim: string
  sentencePairs: SentencePair[]
}
```

## 四、权利要求类型识别

### 4.1 规则预判函数

```typescript
function predictClaimType(text: string): 'method' | 'structure'
```

**方法类信号词**（加权匹配）：
| 信号词 | 权重 | 说明 |
|--------|------|------|
| 步骤 | 3 | 最强信号 |
| 方法 | 2 | 前序部分常见 |
| 流程/过程 | 2 | |
| 当...时/若...则 | 2 | 条件分支 |
| 首先/然后/接着/随后/最后 | 2 | 时序词 |
| 判断/检测/确定/比较 | 1.5 | 判断性动词 |
| 重复/返回/回退 | 1.5 | 循环信号 |

**结构类信号词**（加权匹配）：
| 信号词 | 权重 | 说明 |
|--------|------|------|
| 装置/设备/系统/结构 | 3 | 最强信号 |
| 设置于/位于/连接于 | 2 | 位置关系 |
| 包括/包含/具有/由...组成 | 1.5 | 包含关系（方法类也可能出现） |

**判定规则**：
1. 分别计算方法类总分和结构类总分
2. 分数高者胜出
3. 平局时：检查前序部分（"一种XXX方法" vs "一种XXX装置"），前序部分权重 ×2

### 4.2 AI 确认机制

- 预判结果作为参数传入提示词
- AI 在提取结果中返回 `claimType` 字段
- 如果 AI 返回的 claimType 与预判不一致，以 AI 返回为准
- 前端根据最终 claimType 选择渲染路径

## 五、方法类提示词设计

### 5.1 核心提取规则

#### 5.1.1 节点提取

| 节点类型 | 识别规则 | 视觉形态 | 必须满足的约束 |
|---------|---------|---------|--------------|
| step | 执行性操作/处理步骤 | 圆角矩形（蓝色系） | 至少有一条 sequence 入边或出边 |
| decision | 逻辑判断，产生分支 | 菱形（紫色系） | **必须**有 ≥2 条出边（branch_true + branch_false） |
| condition | 触发条件，不产生分支 | 六边形（橙色系） | 只有 1 条出边（trigger 类型） |

#### 5.1.2 关系提取

| 关系类型 | 识别规则 | 视觉样式 | 约束 |
|---------|---------|---------|------|
| sequence | 时序先后 | 深蓝实线箭头 | source 和 target 不能相同 |
| branch_true | decision 的"是"分支 | 绿色实线箭头，标签"是" | source 必须是 decision 节点 |
| branch_false | decision 的"否"分支 | 红色虚线箭头，标签"否" | source 必须是 decision 节点 |
| trigger | 条件触发 | 橙色点划线箭头 | source 必须是 condition 节点 |
| feedback | 反馈/回路 | 紫色虚线，圆形箭头 | target 的 hierarchyLevel ≤ source |
| parallel | 并行执行 | 青色双线箭头 | source 和 target 不能相同 |
| attribute | 参数/阈值 | 青色虚线，无箭头 | source === target |

#### 5.1.3 decision vs condition 的区分（极其重要）

**decision**（菱形）：
- 语义：**判断**，产生两条或多条互斥路径
- 关键词："判断是否..."、"确定是否..."、"比较...与..."、"检测是否..."
- 出边：branch_true + branch_false（必须同时存在）
- 示例："判断数据是否超过阈值" → [是]→步骤A / [否]→步骤B

**condition**（六边形）：
- 语义：**触发**，满足条件时执行后续步骤，不满足则不执行
- 关键词："当...时"、"在...条件下"、"响应于..."
- 出边：trigger（仅一条）
- 示例："当检测到信号时" → 触发→启动处理

**常见错误**：将"当X时执行Y"误判为 decision。如果条件只触发一个后续步骤（没有"否则"分支），必须用 condition + trigger。

### 5.2 子流程分组

- 如果某个步骤展开为多个子步骤，将子步骤归入同一 group
- group 的 label 为父步骤的名称
- 子流程内的步骤 hierarchyLevel 比父步骤高 1

### 5.3 参数/阈值

- 关键参数值（阈值、时间、数值范围等）用 attribute 类型
- source 和 target 相同，指向拥有该参数的节点
- originalText 填写参数描述（如"阈值=0.5"）
- chineseText 填写中文翻译

### 5.4 首尾步骤标记

- 不使用独立的 start/end 节点
- 流程的第一个 step 节点：绿色加粗左边框
- 流程的最后一个 step 节点：红色加粗右边框
- AI 无需标记首尾，前端根据入边/出边自动判断：
  - 无入边的 step = 首步骤
  - 无出边的 step = 末步骤

## 六、图渲染设计

### 6.1 节点形状

| 节点类型 | X6 Shape | 形状 | 尺寸 | 填充色 | 边框色 | 边框宽度 |
|---------|----------|------|------|--------|--------|---------|
| step | rect（大圆角） | 圆角矩形 | 160×60 | #e8f4fd | #1890FF | 1.5 |
| decision | polygon（菱形） | 菱形 | 140×100 | #f3e8ff | #722ed1 | 2 |
| condition | polygon（六边形） | 六边形 | 160×60 | #fff7e6 | #fa8c16 | 1.5 |

**首尾步骤特殊标记**：
- 首步骤：左边框加粗为 4px，颜色 #52c41a（绿色）
- 末步骤：右边框加粗为 4px，颜色 #e63946（红色）

### 6.2 边样式

| 关系类型 | 颜色 | 线宽 | 线型 | 箭头 | 边标签 |
|---------|------|------|------|------|--------|
| sequence | #1890FF | 4.5 | 实线 | 实心三角 | 无（或步骤描述） |
| branch_true | #52c41a | 4.5 | 实线 | 实心三角 | "是" |
| branch_false | #e63946 | 4.5 | 虚线(6 3) | 实心三角 | "否" |
| trigger | #fa8c16 | 4.5 | 点划线(6 3 2 3) | 空心三角 | 触发条件 |
| feedback | #722ed1 | 3 | 虚线(4 4) | 圆形 | 反馈描述 |
| parallel | #13c2c2 | 4.5 | 实线 | 实心三角 | "并行" |
| attribute | #13c2c2 | 2 | 虚线(3 3) | 无 | 参数描述 |

### 6.3 attribute/参数标签

方法类中 attribute 标签渲染在节点**上方**（而非结构类的底部）：
- stem 边从节点顶部中心连接到标签底部中心
- 标签样式与结构类相同（浅青色背景、青色边框、圆角）

### 6.4 子流程组合框

方法类组合框与结构类组合框结构相同，但样式不同：
- 结构类：橙色虚线框（#fa8c16）
- 方法类：蓝色虚线框（#1890FF），半透明蓝色填充

### 6.5 feedback 边路由

feedback 边需要特殊路由，绕过主流程：
- 从 source 节点底部出发
- 向下延伸到画布底部区域
- 水平移动到 target 节点正下方
- 向上连接到 target 节点底部

实现方式：使用 X6 的 `vertices`（路径点）手动指定路由。

## 七、布局设计

### 7.1 布局方向

- 结构类：自动推断（LR/TB）
- 方法类：**固定 LR**（从左到右），因为流程图天然是时序方向

### 7.2 ELK 配置

方法类专用 ELK 配置：
```typescript
{
  'elk.algorithm': 'layered',
  'elk.direction': 'RIGHT',           // 固定从左到右
  'elk.spacing.nodeNode': '60',
  'elk.layered.spacing.nodeNodeBetweenLayers': '100',
  'elk.layered.crossingMinimization.strategy': 'LAYER_SWEEP',
  'elk.layered.nodePlacement.strategy': 'BRANDES_KOEPF',
  'elk.edgeRouting.orthogonalEdges': 'true',
}
```

### 7.3 feedback 边布局处理

1. 布局时**排除** feedback 边（不传入 ELK）
2. ELK 布局完成后，根据 source/target 节点位置计算 feedback 边的 vertices
3. 手动添加 feedback 边到图中

### 7.4 decision 节点空间

菱形节点比矩形节点需要更多空间，ELK 布局时 decision 节点的 width/height 应适当放大（140×100 vs 160×60）。

## 九、开发注意要点

### 9.1 向后兼容

- 所有新增类型均为**扩展**，不修改现有类型值
- 现有结构类功能不受影响
- GraphJSON 序列化/反序列化需兼容无 claimType 字段的旧数据（默认 'structure'）

### 9.2 高敏感文件

以下文件是本次修改的热点文件，需特别注意增量修改：

| 文件 | 风险 | 注意事项 |
|------|------|---------|
| `src/types/graph.ts` | 中 | 新增枚举值，不修改现有值 |
| `src/types/ai.ts` | 中 | 扩展联合类型，不修改现有字段 |
| `src/services/ai/prompt.ts` | 高 | 新增提示词，不修改现有提示词 |
| `src/services/ai/extractor.ts` | 高 | 扩展验证白名单，不修改现有验证逻辑 |
| `src/services/graph/engine.ts` | **极高** | batchBuild 分支处理，不破坏现有结构类渲染 |
| `src/services/graph/node-builder.ts` | 高 | 新增 shape 支持，不修改现有 rect 节点构建 |
| `src/services/graph/style-registry.ts` | 中 | 新增样式，不修改现有样式 |
| `src/services/graph/layout.ts` | 高 | 方法类布局分支，不修改现有布局逻辑 |

### 9.3 X6 自定义 Shape 注册

- 必须在 `GraphEngine.init()` 之前注册自定义 shape
- 菱形和六边形使用 `Graph.registerNode()` 的 `inherit: 'polygon'`
- polygon shape 的 `refPoints` 定义顶点坐标（归一化到 0-1）
- 节点的 label 需要特殊定位（菱形的文字居中需要调整 refX/refY）

### 9.4 feedback 边的 ELK 兼容性

- ELK 的 cycleBreaking 策略虽然可以处理环路，但会导致 feedback 边被反向
- **必须在布局时排除 feedback 边**，布局后手动添加
- feedback 边的 vertices 计算需要考虑画布的实际尺寸

### 9.5 decision 节点的分支约束

- AI 必须为每个 decision 节点生成 branch_true + branch_false 两条边
- 提取器验证时需检查：如果存在 decision 节点，其出边中必须包含 branch_true 和 branch_false
- 如果 AI 只生成了 branch_true 没有branch_false，应自动补全一条 branch_false 指向"结束"（或报错要求重新提取）

### 9.6 attribute 标签位置

- 结构类：标签在节点底部，stem 从节点底部到标签顶部
- 方法类：标签在节点上方，stem 从节点顶部到标签底部
- 需要在 `renderAttributeTags` 中根据 claimType 选择位置

| 并行处理 | 方法类权利要求可以并行提取 |

### 10.5 边界情况测试

| 场景 | 预期行为 |
|------|---------|
| 空权利要求 | 报错提示 |
| 极短权利要求（1个步骤） | 正常渲染单个 step 节点 |
| 深度嵌套子流程（3+层） | hierarchyLevel 正确，布局不重叠 |
| 多个 feedback 回路 | 每条 feedback 边独立路由，不互相交叉 |
| decision 无 branch_false | 验证报错或自动补全 |
| 全是 condition 无 decision | 正常渲染，所有边为 trigger 类型 |
| 方法类与结构类画布并存 | 互不影响 |
